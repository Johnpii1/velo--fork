# Coding Standards

Velo follows a pragmatic engineering style that favors clarity, explicitness, and maintainability.

## General Principles

- keep functions focused,
- avoid premature abstraction,
- prefer explicit types and interfaces,
- document intent when behavior is subtle,
- keep modules aligned with their responsibilities.

## TypeScript

- use descriptive names,
- prefer `type` over `any` where practical,
- keep route and service logic easy to follow,
- use environment variables for configuration.

## Rust

- keep contract logic auditable,
- avoid unnecessary complexity,
- prefer small, well-named functions,
- document public behavior and assumptions.

## Documentation

Code should be self-explanatory, but non-obvious flows should still be documented in the repository docs or inline comments.
